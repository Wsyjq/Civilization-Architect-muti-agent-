# 开发文档 v2.0：多智能体工作协同与动态评判系统

> 本文档定义新一代游戏机制，从静态数值评判改为**基于工作协同交互的动态评判**，实现真实的 Agent 间多轮工作协作与问题解决。

---

## 一、当前问题分析

### 1.1 现有机制作风

| 问题 | 描述 |
|------|------|
| 评判单一 | 仅通过宏观数值（能量、忠诚度、效率）评判 |
| 对话单向 | Agent A → Agent B 发送消息，无回复互动 |
| 缺乏协同 | 没有多轮工作协调、任务分配、问题解决 |
| 回复相似 | LLM 生成的消息风格单一，缺乏个性化 |
| 无工作流 | 没有明确的工作流程和任务协作 |

### 1.2 期望改进

- **评判基于协同**：通过 Agent 间的工作协作过程和结果评判
- **多轮交互**：Agent 之间针对任务进行多轮讨论协调
- **真实工作流**：包含任务分配、工作协调、问题解决等交互
- **多样性**：每个 Agent 的工作风格符合其性格
- **事件驱动**：每轮有工作事件，交互结果影响产出

## 二、新系统架构

### 2.1 核心概念

```
┌─────────────────────────────────────────────────────────────┐
│                    工作协同循环 (Work Collaboration)            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   工作发布 → 任务认领 → 协作执行 → 问题解决 → 成果评估         │
│                                                             │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐                 │
│   │ 协调者   │ ←→ │ 执行者   │ ←→ │ 协助者   │                 │
│   │ (Leader) │    │ (Worker) │    │ (Helper) │                │
│   └─────────┘    └─────────┘    └─────────┘                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 工作流程

```
Round Start
    │
    ▼
工作发布（Work Publishing）
    │ 系统发布本轮工作任务（如：提高产出、解决危机）
    │
    ▼
任务认领（Task Assignment）
    │ Agent 根据性格和能力认领任务
    │ 高能力者主动承担，高社交者协调
    │
    ▼
协作分组（Collaboration Grouping）
    │ 根据任务需要组成工作组
    │ 3-5 人一组，互补配合
    │
    ▼
多轮工作交互（Multi-turn Work Interaction）
    │ 每轮包含：
    │   1. 方案讨论（多轮来回）
    │   2. 任务协调（分配工作）
    │   3. 问题处理（遇到障碍）
    │   4. 进度同步（汇报进展）
    │
    ▼
成果评估（Performance Evaluation）
    │ 计算各方贡献
    │ 更新宏观变量
    │ 产出分配
    │
    ▼
Round End
```

## 三、工作事件系统

### 3.1 事件类型

| 类型 | 描述 | 示例 |
|------|------|------|
| 常规任务 | 日常生产工作 | "完成本轮生产目标" |
| 突发危机 | 需要立即响应 | "产出下降，需要对策" |
| 资源分配 | 决定资源使用 | "如何分配有限资源" |
| 内鬼暴露 | 发现可疑行为 | "有人行为异常" |
| 协作机会 | 可以协作增效 | "是否联合行动" |

### 3.2 事件生成

事件根据游戏状态动态生成：

```python
事件结构 = {
    "id": "EVENT-001",
    "type": "突发危机",
    "title": "产出急剧下降！",
    "description": "本轮产出比上轮下降了20%，需要找出原因",
    "severity": 0.8,     # 严重程度 0-1
    "required_agents": 3,  # 需要参与的 Agent 数量
    "time_pressure": 0.6  # 时间压力
}
```

### 3.3 事件影响力

事件根据类型影响不同宏观变量：

| 事件类型 | 主要影响变量 |
|----------|-------------|
| 常规任务 | 能级、效率指数 |
| 突发危机 | 向心力、信息保真度 |
| 资源分配 | 能级、社会资本 |
| 内鬼暴露 | 所有变量（负面） |
| 协作机会 | 效率指数、社会资本 |

## 四、Agent 角色与协作

### 4.1 动态角色

Agent 在不同事件中扮演不同角色：

| 角色 | 能力要求 | 行为特征 |
|------|---------|---------|
| 协调者 | 高权威感、高智力 | 分配任务、总结方案 |
| 执行者 | 高效率、高韧性 | 完成分配的工作 |
| 协助者 | 高利他、高社交 | 提供支持、帮助他人 |
| 问题者 | 高智力 | 分析问题、提出方案 |
| 中立者 | 任意 | 观察学习 |

### 4.2 角色分配

根据 Agent 性格和当前状态动态分配：

```
协调者分 = Authority × Intelligence × (1 - current_workload)
执行者分 = Efficiency × Resilience × (1 - busy_level)
协助者分 = Altruism × Sociability × (1 - selfish_level)
```

### 4.3 工作交互类型

| 交互类型 | 描述 | 典型对话 |
|----------|------|---------|
| 方案讨论 | 讨论如何完成任务 | "我认为应该先..." |
| 任务分配 | 分配具体工作 | "你负责这块，我负责..." |
| 进度汇报 | 汇报工作进展 | "我已经完成了..." |
| 问题反馈 | 报告遇到的问题 | "遇到了一些障碍..." |
| 资源请求 | 请求资源支持 | "我需要更多..." |
| 协助请求 | 请求他人帮助 | "能否协助我..." |
| 成果展示 | 展示完成的工作 | "这是我做好的..." |

## 五、多轮工作交互

### 5.1 交互分组

每轮工作交互将 Agent 分为工作组：

| 角色 | 数量 | 说明 |
|------|------|------|
| 核心组 | 3-4人 | 主要工作者 |
| 协助组 | 1-2人 | 提供支持 |
| 观察组 | 0-2人 | 不直接参与 |

### 5.2 多轮工作结构

```
┌────────────────────────────────────────────────────────┐
│                    第一轮：问题分析                        │
├────────────────────────────────────────────────────────┤
│  [分析师] 提出问题："根据数据，问题是原料不足..."         │
│  [协调者] 总结分析："大家怎么看？"                       │
│  [执行者] 补充观点："我认为还有人力分配的问题..."        │
└────────────────────────────────────────────────────────┘
                          ↓
┌────────────────────────────────────────────────────────┐
│                    第二轮：方案讨论                        │
├────────────────────────────────────────────────────────┤
│  [协调者] 提出方案："建议调整资源分配，增加原料..."        │
│  [执行者] 评估方案："方案可行，但需要时间..."            │
│  [协助者] 提出建议："我可以协助加速..."                 │
└────────────────────────────────────────────────────────┘
                          ↓
┌────────────────────────────────────────────────────────┐
│                    第三轮：协调确认                        │
├────────────────────────────────────────────────────────┤
│  [协调者] 确认分工："好，就这么办，你负责A，我负责B..."   │
│  [执行者] 确认接收："明白，我会..."                     │
│  [协助者] 确认协助："我这边没问题..."                   │
└────────────────────────────────────────────────────────┘
```

### 5.3 交互内容生成

**LLM 输入**：
```
当前工作事件：{event_title}
你的角色：{role}（协调者/执行者/协助者）
你的性格：{personality_description}
当前议题：{current_topic}
上下文：{context}
对话历史：{conversation_history}

请生成符合你角色和性格的工作发言。
```

**LLM 输出**：
```json
{
  "message": "生成的工作对话（50-150字）",
  "action_type": "propose/agree/refuse/confirm/request",
  "work_contribution": 0.7,    // 对工作的正面贡献
  "coordination_quality": 0.8,  // 协调质量
  "collaboration_value": 0.6,   // 协作价值
  "task_assignment": {"task": "具体任务", "agent": "负责人"}
}
```

## 六、评判系统

### 6.1 交互质量评估

每轮工作交互结束后，评估整体质量：

```
交互质量 = F(方案可行性, 协调效率, 执行配合度)
```

| 评估维度 | 描述 |
|----------|------|
| 方案可行性 | 提出的方案是否实际可行 |
| 协调效率 | 任务分配是否清晰高效 |
| 执行配合度 | Agent 间配合是否顺畅 |

### 6.2 工作结果对产出的影响

工作交互结果直接影响文明产出：

```
产出计算 = BaseProduction × CollaborationMultiplier

CollaborationMultiplier = F(共识达成度, 方案质量, 参与积极性)
```

| 工作结果 | 产出系数 |
|----------|---------|
| 高效协作，方案通过 | 1.2 - 1.5 |
| 正常完成 | 1.0 |
| 协作困难，部分完成 | 0.7 - 0.9 |
| 完全失败，产出下降 | 0.5 |

### 6.3 Agent 个人贡献评分

工作交互中表现良好的 Agent 获得奖励：

| 行为 | 奖励 |
|------|------|
| 提出可行方案 | +贡献值 |
| 有效协调任务分配 | +贡献值 × 1.5 |
| 积极协助他人 | +贡献值 |
| 解决关键技术问题 | +贡献值 × 2.0 |
| 拖后腿、不配合 | -贡献值 |

```
┌─────────────┐
│ 工作交互结束  │
└──────┬──────┘
       │
       ▼
┌─────────────┐

└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 计算工作结果  │
│ - 方案评分  │
│ - 协作评分  │
│ - 个人贡献  │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 更新宏观变量  │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 计算本轮产出  │
└──────┬──────┘

       ▼
┌─────────────┐
│ 分发贡献奖励  │
```

## 七、内鬼在协作中的行为

### 7.1 内鬼目标

内鬼在工作协作中的目标是：
1. **破坏协作**：制造分歧，阻止共识
2. **抢夺功劳**：将他人成果据为己有
3. **误导方向**：引导协作走向错误方向
4. **拖延时间**：降低协作效率

### 7.2 内鬼策略

| 策略 | 描述 | 表现 |
|------|------|------|
| 搅浑水 | 提出混乱建议 | "我觉得应该...但也不一定..." |
| 抢夺功劳 | 将他人成果归己 | "这正是我想说的..." |
| 误导方向 | 将讨论引向歧途 | "其实问题不在这里..." |
| 拖延时间 | 不做实质贡献 | "让我再想想..." |
| 暗中破坏 | 表面配合实际拆台 | 阳奉阴违 |

### 7.3 内鬼检测

其他 Agent 可以通过工作表现检测内鬼：

```
可疑信号：
- 方案反复无常
- 抢功但不担责
- 表面配合实际拆台
- 与内鬼倾向高的Agent形成小团体
```

## 八、多样性增强

### 8.1 回复多样性

解决当前"回复相似"问题：

1. **性格影响语言风格**
   ```
   高权威感：命令式、果断
   高社交性：热情、询问式
   高私心：利己暗示
   高利他：关心他人
   ```

2. **当前状态影响语气**
   ```
   高压力：紧张、急切
   低效率：沮丧、推脱
   高忠诚：积极、主动
   ```

3. **关系影响表达**
   ```
   高信任：直接、信任
   低信任：委婉、保留
   ```

### 8.2 互动多样性

不同 Agent 组合产生不同互动：

| 组合 | 典型互动 |
|------|---------|
| 领导者 + 老实人 | 高效但缺乏创意 |
| 野心家 + 协调者 | 可能产生冲突 |
| 社交达人 + 分析师 | 协作顺畅 |
| 内鬼 + 高信任者 | 容易被骗 |

## 九、技术实现

### 9.1 新增数据模型

```python
@dataclass
class WorkEvent:
    """工作事件"""
    id: str
    event_type: str  # 常规任务/突发危机/资源分配等
    title: str
    description: str
    severity: float
    required_agents: int
    time_pressure: float

@dataclass
class WorkInteraction:
    """工作交互"""
    id: str
    event_id: str
    round_num: int
    speaker_id: str
    content: str
    action_type: str  # propose/agree/refuse/confirm/request
    work_quality: float

@dataclass
class WorkSession:
    """完整工作会话"""
    event: WorkEvent
    participants: List[Agent]
    coordinator: Agent  # 协调者
    interactions: List[WorkInteraction]
    final_plan: Dict     # 最终方案
    outcome: Dict        # 评估结果
```

### 9.2 LLM 调用优化

每轮工作交互需要多次 LLM 调用：

1. **事件理解**：每个 Agent 理解事件（批量）
2. **方案生成**：工作组生成方案讨论
3. **方案评估**：评估各方案可行性
4. **任务分配**：生成任务分配建议

```
调用次数/工作交互 = 2 + interaction_rounds × participants + 1
```

### 9.3 性能优化

- 批量生成方案（并发调用）
- 缓存已生成的方案（避免重复）
- 简化评估（使用较短的 LLM 调用）
- 工作流异步处理

## 十二、风险与挑战

| 风险 | 描述 | 应对 |
|------|------|------|
| LLM 调用成本 | 每轮多次 API 调用 | 批量并发、缓存 |
| 生成一致性 | 多 Agent 风格差异 | 性格 prompt 优化 |
| 评判公平性 | LLM 评估可能有偏见 | 多维度评估 |
| 计算复杂度 | 实时工作流计算量大 | 异步处理、简化模型 |
